Cube的维护都有可视化界面，业务人员可以像拼魔方一样 修改Cube，

而StarRocks改SQL是快 但是没有可视化界面

# **传统 OLAP（Oracle/IBM）和现代 OLAP（StarRocks/ClickHouse）**在产品设计哲学上的根本分歧

Oracle Essbase / IBM Cognos 是**“全家桶” (All-in-One)。

它们卖给你的不仅是数据库，还有一套极其成熟的、面向业务人员的建模 UI**。业务人员在界面上拖拖拽拽，

就是在维护 Cube。

StarRocks 是**“纯引擎”**。它只关注怎么算得快，它默认你是通过 SQL 来操作它的。

它确实没有原生自带一个给业务人员用的“拼魔方”界面。

# 现在的“业务人员”去哪里拼魔方？

虽然 StarRocks 没有界面，但我们通常会在它上面架设一层工具，让业务人员找回那种“掌控感”。

## 在 BI 工具里建模 (最常见)

如果你用 帆软 FineBI、Tableau、PowerBI 或 DataEase。

以前：业务人员维护 IBM TM1 的 Cube。

现在：业务人员维护 BI 工具里的 “数据集” (Dataset) 或 “语义模型”。

业务人员在 BI 界面里，把 StarRocks 的一张大宽表拖进来。

在 BI 界面里定义：哪个字段是维度（地区），哪个是指标（销售额），要不要重命名，要不要设层级（省->市）。

体验：这和维护 Cube 是一模一样的！也是拖拉拽。

底层：BI 工具会自动把业务人员的拖拽操作，翻译成 SQL 发给 StarRocks 执行。

## 使用“指标平台” (Headless BI)

这是大厂（如美团、字节）现在的玩法。

痛点：BI 工具太多了，有的用 Tableau，有的用 Excel。

解法：在 StarRocks 之上，部署一个指标管理平台（如 Kyligence Zen, Superset 的 Dataset）。

这个平台就是专门给业务/分析师用的“可视化建模工具”。

业务人员在这里定义：“毛利 = 销售 - 成本”。

所有上层应用都调这个平台的接口。

Kyligence (Kylin的商业版) 其实就是把你熟悉的 Cube 建模界面保留了下来，但底层可以用 StarRocks 做加速引擎。

## 这种“分离”带来了什么好处？（为什么要这么折腾？）

既然以前 Oracle 一体化那么好用，为什么现在要拆开？

解耦 (No Vendor Lock-in)：

以前：你的业务逻辑全写在 IBM 的 Cube 里。想换数据库？做梦，重构成本上天。

现在：数据在 StarRocks 里。你的业务逻辑在 BI 软件里。

觉得 StarRocks 不好用？换 ClickHouse，上层 BI 里的报表不用重做。

觉得 BI 软件不好用？换个 BI，底下的 StarRocks 数据不用动。

灵活性：

业务人员在 BI 里建的“虚拟 Cube”是逻辑上的。

今天想加个维度，在 BI 里点一下“刷新字段”就行了。不像以前 Essbase 那样，加个维度要让 IT 停机重构一整天。
