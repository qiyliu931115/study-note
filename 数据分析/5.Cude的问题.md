听起来 Cube 完美无缺，速度快到飞起，那为什么像 StarRocks / ClickHouse 这种新数据库要革它的命？

# 痛点一：改数据 = 牵一发而动全身
场景：发现上个月的一笔订单金额录错了，需要修改。

Essbase / SSAS 的尴尬：

原理：它们是“预计算”机制。底层的一个数字变了，它对应的月度汇总、季度汇总、年度汇总、地区汇总...整条链路上的数据都脏了。

后果：你不能像操作 MySQL 那样 Update 一下就完事。通常必须触发一次**“增量构建”甚至“全量重跑”**。在数据量大时，这意味着你改一个数，可能要等晚上跑批几小时后，报表才能变过来。

IBM TM1 的特权：

这是它唯一的强项。因为它是全内存实时计算的，改了底层的数，上层汇总瞬间自动变。所以它才适合做“财务预算”。

# 痛点二：改逻辑 = 维护者的噩梦
场景：老板说利润计算公式变了，从 收入 - 成本 改成 (收入 - 成本) * 0.9。

IBM TM1 的“规则与喂食”地雷：

Rules（规则）：改公式本身不难。

Feeders（喂食逻辑）：这才是最可怕的。在 TM1 的稀疏内存机制下，改了公式，你必须精准地写代码告诉系统“哪些格子需要被激活”。

后果：Feeders 一旦写错或写漏，轻则数据算不对，重则导致内存瞬间爆炸，服务器直接宕机。

Essbase / SSAS：

需要修改后台的 Calculation Script（计算脚本），然后把海量历史数据重新跑一遍计算。数据越多，耗时越久。

# 痛点三：改结构 = 拆楼重建（最劝退的场景）
场景：公司组织架构调整，或者要在报表里新增一个“用户年龄段”分析维度。

Oracle Essbase 的“物理重构” (Restructure)：

原理：Essbase 的数据在硬盘上是按“稀疏/稠密”维度组合成数据块（Data Block）存储的。加维度意味着物理存储结构变了。

比喻：这就像你盖好了一栋楼，现在非要在每层楼中间硬插一个房间。修修补补是不行的，必须把整栋楼拆了，重新浇筑混凝土。

后果：对于 TB 级的数据，全量重构可能导致系统停机维护一整天甚至更久。

SSAS 的“全量处理” (Process Full)：

原理：修改维度结构后，Cube 会立即变成**“未处理” (Unprocessed)** 状态。

后果：此时用户完全查不到数。管理员必须点击“Process Full”，服务器 CPU 瞬间飙升，疯狂读硬盘重算聚合。在算完之前，整套 BI 系统处于不可用状态。
