# Harness 工程(落地导向)

> 视角:从 Prompt 工程 → Context 工程 → Harness 工程,一条"把 LLM 变成可靠 Agent"的演进主线。
> harness 直译"挽具/马具"——套在马身上、让这股力气能拉车干活的那套绳索缰绳。套在 LLM 外面、让模型这股"智能"能持续、安全、可恢复地干完活的那层**运行时系统**,就是 harness。
> 写法约定:术语后用括号补一句人话;🟢 工程可控　🔴 模型黑箱　💰 主要成本项。
> 更新基准:2026-06。
> 关联文档:速记版脉络见 [HarnessEngineering.md](HarnessEngineering.md);记忆部分见 [长上下文的记忆.md](长上下文的记忆.md);检索部分见 [RAG技术选型.md](RAG技术选型.md);工具协议见 [MCP.md](MCP.md) 与 [MCP和Function Calling区别.md](MCP和Function Calling区别.md)。

## 0. 一句话原则

> **Prompt 决定模型"想得对不对",Context 决定模型"看得到什么",Harness 决定模型"能不能在真实世界里反复把活干完"。**
> 前两者优化的是**单次推理**的质量;Harness 优化的是**多步、长程、带副作用**的执行过程——它管的是模型之外的那一整圈:工具、循环、权限、记忆、子代理、回滚、验证。模型再聪明,没有挽具也只是原地空转的马力。

---

## 1. 三级演进:每一级在补上一级的洞

| 层级 | 核心问题 | 优化对象 | 代表手段 | 它解决不了什么 → 催生下一级 |
|---|---|---|---|---|
| **Prompt 工程** | 模型**听懂**了吗? | 单条指令的表达 | Role/Task/Constraint、Few-shot(给几个示例)、CoT(让它分步想) | 知识缺失、私有数据、长文记忆 → 需要喂对信息 |
| **Context 工程** | 模型**拿到**对的信息了吗? | 进入窗口的内容 | RAG(先检索再作答)、Rerank(重排)、压缩/切片、记忆读回 | 喂对了也只是"答一次";多步执行会跑偏、出错、有副作用 → 需要一套执行系统 |
| **Harness 工程** | 模型在真实执行里能**持续做对**吗? | 模型之外的整个运行时 | 工具调用、控制循环、权限护栏、记忆读写、子代理、验证闭环、回滚 | —(这一层本身就是兜底层) |

> 关键直觉:**前两级把模型当"一个会回答的脑子",Harness 把模型当"一个会行动的员工"。** 员工聪明只是前提,真正决定产出的是:他能用哪些工具、做错了能不能撤销、记不记得上次的结论、越权操作有没有人拦、干完有没有人验收。这些全在模型参数之外,全是工程可控(🟢)的部分。

---

## 2. 什么是 Harness:模型外面那一圈

把一次 Agent 运行拆开,模型本体(那个吐 token 的神经网络)其实只占很小一块。包在它外面、由工程团队完全掌控的那一圈,就是 harness:

```
          ┌─────────────────────────────────────────────┐
          │                  HARNESS                      │
          │  系统提示 / 工具定义 / 权限策略 / 记忆读写      │
          │  控制循环(决定何时再调一次模型、何时停)        │
          │  上下文管理(裁剪、压缩、召回)                  │
          │  子代理调度 / 钩子 / 验证与回滚                 │
          │        ┌───────────────────────┐             │
          │        │   LLM(模型本体)       │  ← 黑箱🔴   │
          │        │   in: context         │             │
          │        │   out: text/tool_call │             │
          │        └───────────────────────┘             │
          └─────────────────────────────────────────────┘
```

> **模型是"决策器",harness 是"决策被执行的整个世界"。** 模型每一步只做一件事:读当前上下文 → 输出一段文字或一个工具调用。要不要真的执行这个调用、执行结果怎么塞回上下文、要不要再问一次模型、什么时候判定"干完了"——全由 harness 决定。模型不控制循环,**harness 才是那个 while 循环**。

## 3. Harness 的核心组成(逐个拆)

### 3.1 工具(Tools)——给模型装上手
* 模型本身只会"说话",不会"做事"。工具把"做事"的能力以函数签名的形式暴露给模型(读文件、跑命令、查数据库、发请求)。
* 工程要点:**工具的描述就是给模型的提示词**。命名、参数说明、什么时候该用/不该用,直接决定模型用得对不对。好的工具描述会写明"优先用 X 而不是 Y""这是只读还是有副作用"。
* 协议层:Function Calling 是模型原生的"我要调这个函数"约定;MCP(Model Context Protocol)是把一批工具/数据源标准化对接进 harness 的协议。区别见 [MCP和Function Calling区别.md](MCP和Function Calling区别.md)。
* 反模式:工具堆太多 → 模型选择困难、上下文被工具定义撑爆。成熟 harness 会**按需加载工具定义**(用到才把 schema 喂进去),而不是一次性全摆出来。

### 3.2 控制循环(Agent Loop)——决定"再想一步还是收工"
* 最朴素的形态:`模型输出 → 若是工具调用则执行 → 把结果拼回上下文 → 再问模型 → ……直到模型说"做完了"`。
* 工程要点:**停机条件**(什么时候判定完成)、**步数上限**(防止无限打转)、**错误是反馈回去让它自纠还是直接中断**。
* 一个工具调用被拒绝(权限不允许),不该傻傻原样重试 → harness 要把"被拒"作为信号喂回,让模型换方案。

### 3.3 上下文管理——窗口是稀缺资源,要主动经营
* 长程任务里,历史会越滚越长 → 成本线性涨 + 中段遗忘(Lost in the Middle,内容堆太多时正中间的信息易被忽略)。详见 [长上下文的记忆.md](长上下文的记忆.md)。
* harness 的职责:**裁剪**(丢掉不再相关的旧内容)、**压缩/总结**(把一大段历史折叠成摘要,即 compaction)、**召回**(需要时再把外部记忆检索回来)。
* 经典手法:当对话快撑满窗口,harness 自动把前文压成摘要,腾出空间继续——任务不必中断。

### 3.4 记忆(Memory)——窗口之外的长期读写
* 工作记忆(在窗口里)≠ 长期记忆(在窗口外)。长期记忆必须落到数据库/向量库/文件,每次对话**按需读回**,本质是 RAG。
* harness 决定:什么值得记(用户偏好、过往结论)、记成什么结构(一条事实一个文件、带元数据)、何时读回。
* 这一整套取舍是 [长上下文的记忆.md](长上下文的记忆.md) 的主题。

### 3.5 子代理(Sub-agents)——分而治之 + 隔离上下文
* 把一个大任务拆给若干个"干净开局"的子代理并行做,各自带独立上下文,结果汇总回主代理。
* 好处:**上下文隔离**(子代理搜一堆文件的噪音不污染主线)、并行提速、专项分工(搜索代理、评审代理)。
* 代价💰:每个子代理冷启动、重新建立上下文,token 开销不小 → 只在真正独立、可并行的子任务上用,不滥用。

### 3.6 权限与护栏(Permissions / Guardrails)——可控副作用
* Agent 会真的改文件、跑命令、调外部 API,这些是**不可逆副作用**。harness 必须在"模型想做"和"真的执行"之间插一道闸。
* 形态:权限模式(只读/需确认/放行)、危险操作(删除、推送、转账、对外发布)强制人工确认、敏感工具按场景授权。
* 原则:**一个场景的授权不自动延续到下一个场景**;难以撤销或对外可见的动作,默认先确认。

### 3.7 钩子(Hooks)——把"每次都要做的事"固化成规则
* 模型有时不稳定执行"每次都……"的要求。把这类硬规则放进 harness 的钩子(在某动作前/后自动触发的脚本),由系统强制执行,而不是指望模型记得。
* 例:每次提交前自动跑 lint;某工具调用后自动校验输出格式。这是"工程兜底模型不确定性"的典型。

### 3.8 技能 / 程序化知识(Skills)——可复用的工作流封装
* 把"做某类任务的标准步骤"封装成可调用的技能包,需要时加载其完整说明,模型照着做。
* 价值:把专家流程沉淀成 harness 资产,跨会话复用,不必每次在 prompt 里重述。

### 3.9 验证闭环(Verification)——干完不等于干对
* 最容易被忽略、也最关键的一环:**断言"完成"之前,先跑验证拿到证据**(跑测试、跑起来看输出),而不是凭模型自信。
* 失败要如实回报(测试挂了就说挂了),而不是模型嘴上说"已修复"。这是把 Agent 从"看起来能用"推向"真的可靠"的分水岭。

## 4. 一手样本:这篇文档就是在一个 harness 里写出来的

> 我(写这篇笔记的 AI)此刻就跑在一个成熟 harness 内。对照第 3 节,几乎每一项都能在我的运行环境里找到实物:

| 第 3 节组成 | 在我当前运行环境里的对应物 |
|---|---|
| 工具 | 读/写/编辑文件、跑命令、搜索代码等一组带说明的函数 |
| 控制循环 | 我每轮只输出"文字或工具调用",由外层系统执行并把结果回灌,再触发我下一轮 |
| 上下文管理 | 对话变长时系统会把前文**压成摘要**再续上,我不必草草收尾 |
| 记忆 | 一个磁盘上的记忆目录:一条事实一个文件 + 一个索引,跨会话读回 |
| 子代理 | 可派发只读搜索代理、代码评审代理并行干活,结果回汇 |
| 权限与护栏 | 工具调用跑在"权限模式"下;难逆/对外的动作要先确认 |
| 钩子 | 某些动作前后被系统拦截,钩子输出当作反馈对待 |
| 技能 | 一批可按需加载的 skill,触发某类任务时先读它的完整流程 |
| 验证闭环 | 规则明确要求:声称"完成/已修复"前必须先跑验证拿证据 |

> 这正说明:**Harness 工程不是一套理论,而是已经在产品里跑着的工程实践。** 同一个底层模型,套上更好的 harness,就能从"会聊天"变成"能交付"。模型能力是上限,harness 决定你实际拿到上限的百分之几。

## 5. 设计原则与工程取舍

| 原则 | 说明 |
|---|---|
| **模型负责决策,harness 负责执行与兜底** | 凡是"必须每次都对"的(权限、验证、格式),交给确定性的工程,别赌模型每次都记得。 |
| **上下文是预算,不是免费仓库** | 每一段塞进窗口的内容都在花💰、都在稀释注意力。主动裁剪/压缩,而非无脑堆。 |
| **副作用要可控、可回滚** | 改动越难撤销,闸门越要前置(确认、沙箱、worktree 隔离)。 |
| **失败是一等公民** | 设计错误反馈通路:被拒、报错、超时都要能喂回让模型换路,而非崩或硬重试。 |
| **隔离换清晰** | 噪音大的子任务丢给子代理,保主线上下文干净。 |
| **证据先于断言** | 完成主张必须有验证输出支撑。 |

> 核心张力:**模型的"灵活/不确定" vs 工程的"可靠/确定"**。Harness 工程的本质,就是在两者之间划线——把不能容错的部分硬编码进系统,把需要灵活判断的部分留给模型。划得越准,Agent 越可靠。

## 6. 评估与可靠性(让 harness 可迭代)

* **Eval(自动化评估)**:用基准任务集持续跑,监控模型/harness 改动后是否退化。没有 eval,harness 的每次改动都是盲改。
* **Self-Correction(自纠错)**:用 Reflection 机制让模型回看自己的输出再修一版——但要注意,自纠错不能替代外部验证(模型可能自信地错)。
* **可观测性**:每一步工具调用、上下文变化、决策路径可追溯,出问题能定位是模型判断错还是 harness 执行错。

## 7. 常见误区

| 误区 | 纠正 |
|---|---|
| "换个更强的模型就行了" | 模型是上限,harness 决定你拿到多少。弱 harness 会把强模型的能力大量漏掉。 |
| "Prompt 写够好就不需要 harness" | Prompt 管单次表达,管不了多步执行的循环、副作用、回滚。 |
| "上下文窗口够大就把全部历史塞进去" | 成本线性涨 + 中段遗忘,反而更差。要主动经营窗口。见 [长上下文的记忆.md](长上下文的记忆.md)。 |
| "工具越多越能干" | 工具过载会让模型选择困难、撑爆上下文。按需加载、精选工具。 |
| "模型说做完了就是做完了" | 没有验证证据的"完成"不可信。证据先于断言。 |
| "权限交给模型自己判断" | 不可逆副作用必须有确定性护栏,不能赌模型每次都谨慎。 |

## 8. 一句话收束

> **Prompt 工程教模型怎么想,Context 工程喂模型该看什么,Harness 工程则建造一个让模型能反复、安全、可恢复地把事做完的世界。**
> 行业重心正从"调教单次对话"转向"建造可靠的执行系统"——因为决定一个 Agent 好不好用的,越来越不是模型本身,而是它外面那套挽具。
