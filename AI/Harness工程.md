# Harness 工程

讨论的是同一条主线:怎么把一个聪明的模型,做成一个能可靠干完多步活的智能体(Agent)。

harness 直译是"挽具"(套在牲口身上拉车那套绳索),这里指**套在大模型外面、让它能持续、安全、可恢复地完成任务的那层运行时系统**。模型负责出主意,harness 负责让这些主意真的被安全执行。

---

# 一个公式:Agent = Model + Harness

如果这篇只记一句话,就记这个。

模型(Model)负责推理,harness 负责**其余的一切**:工具、上下文管理、权限控制、反馈回路、记忆、协作。

一个好用的类比:**模型是 CPU,harness 是操作系统**。CPU 再快,操作系统拉胯,整机也跑不快——这个比喻对应的是"算力"和"调度环境"的分工关系,删掉它你就不容易理解为什么"换更强的模型"经常不是瓶颈所在。边界也要说清:这个类比只到分工为止,别据此以为"换 harness"能像换操作系统那样即插即用——harness 和具体模型的配合往往是要一起调的。

提出这个视角的是 Mitchell Hashimoto(HashiCorp 联合创始人,开源终端 Ghostty 的作者)。他给 harness 工程下的定义很朴素:

> 每当你发现 Agent 犯了一个错,就花时间工程化一个方案,让它将来不再犯同样的错。

也就是说,**Agent 的失败往往不是模型不够聪明,而是它的运行环境没设计好**。

有个能查到的实证:LangChain 在 Terminal Bench 2.0(一个公开的终端任务评测基准)上,**完全不换模型**,只优化 harness(加上自我验证、过程追踪、文档),排名从第 30 升到第 5,分数从 52.8 涨到 66.5。这说明在不少场景里,优化模型之外的运行环境,回报比换一个更强的模型更高。

# 三代演进:Prompt → Context → Harness

| 代际 | 范式 | 核心问题 | 解决不了什么 |
|---|---|---|---|
| 一 | 提示词工程(Prompt Engineering) | 怎么把话说清楚 | 知识缺失、私有数据、长文记忆 |
| 二 | 上下文工程(Context Engineering) | 怎么给模型喂对信息 | 喂对了也只是"答一次",多步执行会跑偏、出错、有副作用 |
| 三 | 驾驭工程(Harness Engineering) | 怎么让 Agent 可控地干活 | —— |

区别在于看待模型的方式变了:**前两代把模型当成"一个会回答问题的脑子",第三代把它当成"一个会动手干活的执行者"**。

当它会动手了,决定产出的就不只是它多聪明,还包括:它能用哪些工具、做错了能不能撤销、记不记得上次干到哪、越权操作有没有人拦、干完有没有人验收。这些都在模型参数之外,都是工程能控制的部分。

---

# Harness 到底是什么:模型外面那一圈

把一次 Agent 运行拆开看,模型本体(那个吐字的神经网络)其实只占很小一块。包在它外面、由工程完全掌控的那一圈,就是 harness:

```
          ┌─────────────────────────────────────────────┐
          │                  HARNESS                      │
          │  系统提示 / 工具定义 / 权限策略 / 记忆读写      │
          │  控制循环(决定何时再问一次模型、何时收工)      │
          │  上下文管理(裁剪、压缩、召回)                  │
          │  子代理调度 / 钩子 / 验证与回滚                 │
          │        ┌───────────────────────┐             │
          │        │   模型(只负责出主意)  │             │
          │        │   读 context          │             │
          │        │   吐 文字 / 工具调用   │             │
          │        └───────────────────────┘             │
          └─────────────────────────────────────────────┘
```

关键一点容易被忽略:**那个"反复调用模型直到完成"的循环,写在 harness 里,不在模型里**。模型每一步只做一件事——读当前上下文,吐出一段文字或一个工具调用。要不要真执行、结果怎么塞回去、要不要再问一次、什么时候算干完,全是 harness 说了算。模型是决策器,harness 是决策被执行的整个环境。

从系统层面看,它大致是五层叠起来的:**用户交互层 → 编排层 → 能力层 → 执行层 → 对外接口(MCP)**。其中**工作区(Workspace)** 横跨各层,是所有层共享的状态存放地。有四类约束垂直穿过这五层:上下文工程、架构约束、反馈回路、熵管理(entropy,这里指系统随时间增长的混乱度)。做得好的系统,不是某一层特别花哨,而是这四类约束在每一层都有落点。

有个值得记住的说法:**Agent 系统正在长成一台新的计算机**——交互层是它的 Shell,编排层是调度器,能力层是系统调用,工作区是文件系统,MCP 是设备驱动。

# 四条反直觉的设计原则

这四条都跟工程师的直觉相反,所以最容易踩坑。

**一、上下文越少越好,不是越多越好。**
直觉是"信息越多,决策越准"。但上下文是稀缺资源,塞太多会互相干扰,也会让模型在一堆选项里来回游移、忘了正事。长上下文窗口被填到一定比例后,准确率会明显下降(具体阈值因模型、任务、提示结构而异,没有一个通用百分比)。所以"模型支持 200K" 不等于"可以塞 200K"——给 Agent 的上下文,要像做 Code Review 一样一条条挑。

**二、专用 Agent 优于通用 Agent。**
一个什么都管的"全能 Agent",会在几十个工具的列表里频繁挑错工具,效果反而不如一组各管一摊的专用 Agent。前面 LangChain 那次评测里,**把任务拆给专门的子代理(Sub-agent)、每个只装它真正用得上的工具**,就是排名从 30 冲到 5 的关键改动之一。一条经验法则:**Agent 是贵的,技能(Skill)是便宜的**,能用一个 Skill 解决的,就别新增一个 Agent。

**三、状态写文件,不要塞进上下文。**
上下文是易失的(会话一结束就没了),文件系统才是持久内存。任务的中间结果、跨会话的延续、事后的审计回放,都应该走文件。把工作区当成 Agent 的版本库:每一步都可回放、可审计、可断点续传。**上下文窗口只是"当前工位",工作区才是"档案柜"。**

**四、能写成检查规则的约束,就别只写进文档。**
文档对模型只是"建议",而 Linter / CI 才是"强制"。自然语言会被模型自由解读,代码不会。把规则编码成机器能执行的检查(静态检查、CI 拦截),比再写一段更详细的提示词可靠得多。Mitchell Hashimoto 在自己的项目里就是这么做的:规则文件里每一条都对应一个真实出过的错,凡是能转成自动检查的,绝不让它停留在自然语言。

# 常见的工程做法

前面讲的是原则,这里是几种具体做法——它们在不少团队的实践里都出现过,可以参考照搬。

1. **双阶段:先规划再执行。** 一个 Agent 负责理解任务、写出 `plan.md` 然后退出;另一个 Agent 读 `plan.md` 按步执行。两者不共享上下文,只靠工作区里的这份计划接力,于是任务可以跨多次会话延续,不依赖任何一次会话的记忆。

2. **工具签名就是文档。** 工具名用动词短语(`query_calendar` 而不是 `tool_03`);每个参数的说明都写清"什么时候用、什么时候别用";返回结构稳定,模型不用每次猜格式。Agent 选错工具,多半不是工具太多,而是工具签名写得像一团糨糊。

3. **子代理之间互相隔离。** 重点不在"拆",在"隔离":每个子代理有独立上下文、只看到自己那几个工具;主 Agent 只接收它的结构化结果,不接收它一路上的中间思考(那些是噪音)。

4. **上下游反压(Backpressure)。** 上游给确定的配置,Agent 执行,下游用测试 / 类型检查 / Lint / CI 把不合格的结果挡回去,错误信号回传上游再调整。这里有个细节:**Linter 的报错本身就是一种上下文工程**——它不只说"违反了规则 X",还顺带解释"为什么有这条规则、正确做法是什么",这样模型读完报错就能自己改对,不用人介入。

5. **让一个 Agent 去审另一个 Agent。** 人审不过来时,用另一个 Agent 做 Code Review,但必须换上下文:审查方只看 `git diff` 和规则文档,设定成一个持怀疑态度的评审者,对原 Agent 干活的过程一无所知,才不会被"自我合理化"带偏。**用同一个上下文再评一遍,只会把同一个偏见复现一次。**

6. **熵管理:派个 Agent 当园丁。** 代码库的混乱会随时间增长,而 **Agent 往往比人长得更快**——它擅长复制模式,也就会忠实地复制并放大坏模式。办法是让一个后台 Agent 定期扫过期文档、检测架构漂移、提清理 PR,把"垃圾回收"做成定时任务,而不是攒到项目末尾再收拾。

# Harness 里都有哪些零件

把上面散落的部件归一下类,一个完整的 harness 大致由这些组成:

- **工具(Tools)**:给模型可以动手的能力。工具描述本身就是提示词;schema 按需加载,别一次性全摆出来撑爆上下文。
- **控制循环(Loop)**:决定"再想一步还是收工"。要有停机条件和步数上限;被拒绝或报错时,把它当信号反馈回去让模型换路子,而不是傻傻原样重试。
- **上下文管理**:经营窗口这块预算——裁剪、压缩(compaction,把一大段历史折叠成摘要)、按需召回。
- **记忆(Memory)**:窗口之外的长期读写。一条事实存一处加索引,每次按需读回,本质就是 RAG。
- **子代理(Sub-agent)**:分解任务、隔离上下文。各带独立上下文和专属工具,只回汇结构化结果;冷启动有成本,别滥用。
- **权限控制**:约束副作用。只读 / 需确认 / 放行分级;难撤销或对外可见的动作,默认先确认。
- **钩子(Hooks)**:把"每次都得做的事"固化成由系统强制执行的脚本(比如提交前自动跑 lint),不指望模型每次都记得。
- **技能(Skills)**:把可复用的标准流程封装起来,跨会话复用。
- **验证**:区分"干完了"和"干对了"。声称完成之前先跑验证拿到证据;失败要如实报,不能嘴上说"已修复"。

# 这篇文档本身就跑在一个 harness 里

我(写这篇笔记的 AI)此刻就运行在一个挺成熟的 harness 内,上面讲的原则和做法,大多能在我的运行环境里找到对应的实物:

- **上下文越少越好** → 对话变长时,系统会自动把前文压成摘要再续上(compaction),不会无脑保留全部历史。
- **专用优于通用** → 我可以派出只读搜索代理、代码评审代理这种专用子代理,各自并行干活。
- **状态写文件** → 我有一个磁盘上的记忆目录:一条事实一个文件,外加一个索引,跨会话读回。
- **约束写成检查规则** → 某些动作前后会被钩子拦截,钩子的输出我当作反馈对待——硬规则由系统强制,不靠我自觉。
- **反压** → 工具调用被拒或报错,会作为信号回灌给我,让我换方案而不是硬重试。
- **验证** → 规则明确要求:声称"完成 / 已修复"之前,必须先跑验证拿到证据。

所以 harness 工程不是纸上谈兵,它已经实实在在地跑在产品里了。同一个底层模型,套上更好的 harness,实际能发挥出来的能力差别很大。**模型能力决定上限,harness 决定你实际拿到上限的几成。**

# 一个真实案例:阿里的 AI 招聘项目

下面这个例子来自阿里在钉钉上落地的一个 AI 招聘项目。数字都是它内部实测,只作"改造前后"的相对参考,不是行业基准。

**第一版,做了个全能招聘 Agent。** 一份 600 多行的 System Prompt 加 13 个工具,招聘流程全包。结果问题全冒出来了:RPA(机器人流程自动化)跑到一半"顺手"去回复候选人聊天,上下文乱套;工具一多,挑错工具的概率明显偏高(该查日历的调成了查库);提示词太长,写到后面模型已经忘了开头那句"不要主动发 Offer";状态全在上下文里,换个会话就不记得这个候选人了;出错也难定位,回滚一次得翻好几个小时日志。这一版同时踩了原则二(专用优于通用)和原则一(上下文越少越好)。

**第二版,拆成"2 个 Agent + 一堆 Skill"。**

```
编排层(拆任务 / 分发 / 维护工作区 / 跨会话记忆)
  ├── 人岗匹配 Agent(管 RPA,Prompt 80 行,4 个工具)
  └── 招聘沟通 Agent(管聊天,Prompt 90 行,5 个工具)
        └── 一批原子化 Skill(解析简历 / 打分 / 查日历 / 订会议室 ...)
              └── 工作区(状态来源:/candidates/{id}/state.json 等)
```

最关键的一招:**编排层自己不直接调那一堆工具,只调"派单"这一个动作**——把任务连同精挑出的 3 到 5 个工具,派给对应的专用 Agent。这就把"几十里选一"的难题,拆成了几道"五里选一"。

四条原则各自的落点也很清楚:每个 Agent 的提示词压在 100 行以内(原则一);整个系统只留 2 个 Agent,其余全部下沉成 Skill(原则二);候选人状态、RPA 进度都写进工作区(原则三);合规规则(像"不许主动发 Offer""超时 72 小时必须提醒 HR")写成检查规则,工具调用前先校验,不靠提示词自觉(原则四)。

改造之后:工具选错率明显下降,端到端准确率稳定超过上线门槛,日志按 Agent 分桶、能分钟级定位问题,上下文消耗回到安全区,来了新需求多数情况下加一个 Skill 就行。

这个案例还沉淀出三条经验:

- **他们的经验是:Agent 数量通常控制在 3 个以内,Skill 可以随便加。** 一度想再拆出"日报 Agent""超时预警 Agent",堆到第 6 个时,编排层自己开始选错 Agent 了——Agent 的数量本身也是一种上下文成本。后来把这些全改成 Skill,问题就消失了。这条是他们这个场景里的观察,不是普适上限。
- **RPA 和 Agent 的接缝最容易出事,要做"事务边界"。** 做法是引入一个锁文件 `rpa_lock/{batch}.json`,记 running / done,中途断了下次从断点续传,绝不从头重跑。这正对应原则三——状态写文件,不放脑子里。
- **对外说话的 Agent 必须套硬约束。** 招聘沟通 Agent 是唯一会主动给真人发消息的,它的错是直接暴露在外的。给它套了三层:① 白名单工具(只能发消息,禁用撤回 / 群发);② 所有外发先过敏感词和合规检查,过不了直接拒掉;③ 再让另一个 Agent 用独立上下文审稿。三层叠上去,对外消息的事故率明显降了下来。

# 顺便看看别家怎么做

不是排名,只是同样那几类约束,落在不同系统里的位置不一样:

- **Claude Code(Anthropic)**:用"先规划再执行"的双阶段,把工作区当持久层——跨会话延续是实打实的生产力,不是噱头。
- **LangChain**:靠自我验证 + 过程追踪 + 工具签名优化,不换模型就冲进评测前五——优化 harness 的回报常常比换模型更高。
- **Ghostty(Mitchell Hashimoto)**:规则文件当"项目宪法",每条规则背后都有一个真实失败——能机器化的约束,就别留在自然语言里。
- **Cursor / Cline**:内置反馈回路(Linter / 类型检查 / 测试)自动闭环——反馈回路本身就是上下文,错误信息要写得让模型看得懂、改得动。

# 几个容易踩的坑

- **"换个更强的模型就行了"**:模型是上限,harness 决定你实际能用到多少。前面那个不换模型、只改 harness 就从 30 升到 5 的例子,就是反例。
- **"提示词写够好就不用 harness"**:提示词管的是单次表达,管不了多步执行里的循环、副作用和回滚。
- **"窗口够大就把全部历史都塞进去"**:成本会线性上涨,还有中段遗忘(Lost in the Middle,内容堆太多时正中间的信息最容易被忽略),效果反而更差。
- **"工具越多越能干"**:工具一过载,模型就开始挑错、上下文也被撑爆。按需加载、精挑工具。
- **"做一个全能 Agent 全包"**:通用干不过专用。Agent 贵、Skill 便宜,能用 Skill 就别加 Agent。
- **"让 Agent 自己记住进展"**:上下文是易失的,状态要落到文件,工作区才是档案。
- **"把规则写进文档让它自己读"**:文档是建议,Linter / CI 才是强制。能机器化就别留自然语言。
- **"模型说做完了就是做完了"**:没有验证证据的"完成"不可信,证据先于结论。

# 小结

- 你优化的不是 Agent 本身,而是 Agent 的工作环境。
- 上下文是稀缺资源,干净比丰富更重要。
- 状态写在文件里,不放在上下文里。
- 能写成检查规则的约束,优先于写进文档。
- Agent 比 Skill 贵,Skill 比纯约束贵,能用便宜手段解决就别上贵的。
- 对外说话、动用户数据的环节,硬约束要尽早到位。

一句话:模型能力决定上限,harness 决定你实际拿到多少。这两年行业的重心,正从"调一次对话"转向"建一套能可靠干活的执行系统",而工程能力的差距,也正在从这里拉开。
