# Harness 工程(给野马套上缰绳)

> 视角:从 Prompt 工程 → Context 工程 → Harness 工程,一条"把 LLM 从聪明的脑子变成可靠的员工"的演进主线。
> harness 直译"挽具/缰绳"——套在马身上、让这股蛮力能拉车干活的那套绳索。套在 LLM 外面、让模型这股"智能"能持续、安全、可恢复地干完活的那层**运行时系统**,就是 harness。模型再聪明,没有缰绳也只是一匹能跑也能毁的野马。
> 写法约定:术语后用括号补一句人话;🟢 工程可控(确定性)　🔴 模型黑箱(不确定性)　💰 主要成本项。
> 更新基准:2026-06。
> 关联文档:速记版脉络见 [HarnessEngineering.md](HarnessEngineering.md);记忆部分见 [长上下文的记忆.md](长上下文的记忆.md);检索部分见 [RAG技术选型.md](RAG技术选型.md);工具协议见 [MCP.md](MCP.md) 与 [MCP和Function Calling区别.md](MCP和Function Calling区别.md)。
> 主要参考:阿里技术《给野马套上缰绳:Agent Harness 工程实践》;一手来源 Mitchell Hashimoto《My AI Adoption Journey》、LangChain《Improving Deep Agents with Harness Engineering》。

## 0. 一句话原则

> **Agent = Model + Harness。** 模型负责推理,Harness 负责"剩下的所有事":工具、上下文、权限、反馈回路、记忆与协作。
> Prompt 决定模型"想得对不对",Context 决定模型"看得到什么",Harness 决定模型"能不能在真实世界里反复把活干完"。前两者优化**单次推理**,Harness 优化**多步、长程、带副作用**的执行过程。
> 一句更狠的话(Mitchell Hashimoto):**每当你发现 Agent 犯了一个错,就花时间工程化一个解,让它将来不再犯同样的错。** ——Agent 的每一次失败,都不是模型的问题,而是环境设计欠的债。

---

## 1. 缘起与定义:这个词从哪来

* **谁提的**:Terraform / HashiCorp 联合创始人、Ghostty 终端作者 **Mitchell Hashimoto** 在个人博客《My AI Adoption Journey》中系统提出 "Engineer the Harness"。
* **原话定义**:*"anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent will not make that mistake again in the future."*
* **核心公式**:`Agent = Model + Harness`——被 LangChain 在《Improving Deep Agents with Harness Engineering》提到标题级论断。模型推理,Harness 负责工具系统、上下文管理、权限控制、反馈回路、记忆与协作。
* **一个类比**:**模型是 CPU,Harness 是操作系统(OS)**。CPU 再快,OS 拉胯也白搭。Anthropic 的 Claude Code 能撑起商业化重要一极,本质不是更强的模型,而是更好的 Harness。

> **一手实证(最该记住的案例)**:LangChain 在 Terminal Bench 2.0 上**不换模型**,只优化 Harness(自我验证 + 追踪 + 文档),排行榜从**第 30 冲到第 5**,得分 **52.8 → 66.5**。
> 结论:过去几年大量精力放在"换更强的模型"上,但**真正的杠杆一直在模型之外**。Harness 的回报曲线,比换模型陡得多。

## 2. 三代范式演进:你站在第几代?

| 代际 | 范式 | 核心问题 | 形象比喻 | 它解决不了什么 → 催生下一代 |
|---|---|---|---|---|
| 第一代 | **Prompt 工程** | 怎么把话**说清楚** | 对马**喊话**的技巧 | 知识缺失、私有数据、长文记忆 → 需要喂对信息 |
| 第二代 | **Context 工程** | 怎么给 AI **喂对信息** | 给马看的**地图** | 喂对了也只是"答一次";多步执行会跑偏、出错、有副作用 → 需要执行系统 |
| 第三代 | **Harness 工程** | 怎么让 Agent **可控地工作** | 给马造**高速公路**,配护栏、限速牌、加油站 | —(这一层本身就是兜底层) |

> 关键直觉:**前两代把模型当"一个会回答的脑子",Harness 把模型当"一个会行动的员工"。** 员工聪明只是前提,真正决定产出的是:他能用哪些工具、做错了能不能撤销、记不记得上次干了什么、越权操作有没有人拦、干完有没有人验收。这些全在模型参数之外,全是工程可控(🟢)的部分。

## 3. 什么是 Harness:模型外面那一圈

把一次 Agent 运行拆开,模型本体(那个吐 token 的神经网络)其实只占很小一块。包在它外面、由工程团队完全掌控的那一圈,就是 harness:

```
          ┌─────────────────────────────────────────────┐
          │                  HARNESS                      │
          │  系统提示 / 工具定义 / 权限策略 / 记忆读写      │
          │  控制循环(决定何时再调一次模型、何时停)        │
          │  上下文管理(裁剪、压缩、召回)                  │
          │  子代理调度 / 钩子 / 验证与回滚                 │
          │        ┌───────────────────────┐             │
          │        │   LLM(模型本体)       │  ← 黑箱🔴   │
          │        │   in: context         │             │
          │        │   out: text/tool_call │             │
          │        └───────────────────────┘             │
          └─────────────────────────────────────────────┘
```

> **模型是"决策器",harness 是"决策被执行的整个世界"。** 模型每一步只做一件事:读当前上下文 → 输出一段文字或一个工具调用。要不要真的执行、结果怎么塞回上下文、要不要再问一次模型、什么时候判定"干完了"——全由 harness 决定。模型不控制循环,**harness 才是那个 while 循环**。

### 五层运行时 + 四根护栏(系统视角)
把三代范式落到系统层面,本质是一张"围绕 Model 展开的运行时全景图":

* **纵向五层**:`用户交互层 → 编排层 → 能力层 → 执行层 → MCP(对外接口)`,是一次用户请求在系统里流过的完整路径。**Workspace(工作区)** 横跨各层,是所有层共享的**状态基座**(真相之源)。
* **左侧四根护栏**(垂直穿透每一层):**Context Engineering(上下文工程)/ Architecture Constraints(架构约束)/ Feedback Loop(反馈回路)/ Entropy Management(熵管理)**。成熟团队不是某一层做得花哨,而是这四根护栏在五层里**都有落点**。

> 一个值得反复咀嚼的判断:**Agent 系统是一种新的计算操作系统**——用户交互层是它的 Shell,编排层是 Scheduler,能力层是系统调用,Workspace 是文件系统,MCP 是设备驱动。

## 4. 四条反直觉铁律(心法)

> 之所以"反直觉",是因为它们都和工程师的本能背道而驰——而这正是它们值钱的原因。

| 铁律 | 工程师本能 | Harness 真相 |
|---|---|---|
| **一** | 信息越多决策越准 | **上下文越少越好**;上下文是稀缺资源,要像 Code Review 一样精挑 |
| **二** | 一个超级 Agent 全包 | **专才 Agent 永远赢过通才 Agent** |
| **三** | 让 Agent "记住"任务进展 | **状态要写文件**,不要塞上下文 |
| **四** | 把规则写进 AGENTS.md 读给它 | **能写成 Linter 的约束,别停留在文档** |

* **铁律一 · 上下文越少越好**:长上下文窗口的使用比例超过某阈值后,准确率会显著下降(阈值因模型/任务而异,没有放之四海皆准的百分比)。**"模型支持 200K" ≠ "可以塞 200K"**。详见 [长上下文的记忆.md](长上下文的记忆.md) 的"中段遗忘(Lost in the Middle)"。
* **铁律二 · 专才 > 通才**:通才 Agent 在工具列表里"逛超市",跑不过一组职责清晰的专才。LangChain 实测:把任务拆给专门的 Sub-Agent、每个只装它真正需要的工具,是排名 30→5 的关键改动之一。推论:**Agent 是昂贵的,Skill 是廉价的——能用 Skill 解决就别新增 Agent。**
* **铁律三 · 状态写文件**:上下文是易失存储(关掉就没),文件系统才是持久内存。**Workspace 是真相,Context Window 只是当前工位**。中间结果、跨会话延续、审计回放全走文件系统——把 Workspace 当成"Agent 的 Git 仓库",可回放、可审计、可断点续传。
* **铁律四 · 约束可执行**:文档只是"建议",Linter / CI 才是"强制"。自然语言会被模型"创造性解读",代码不能。**把规则编码成机器可执行的约束,比"再写一段更详细的 Prompt"可靠十倍。**

## 5. 六大工程模式(招式:可直接抄的设计模式手册)

| # | 模式 | 解决的核心问题 |
|---|---|---|
| 1 | **双阶段架构(Init + Exec)** | 跨会话延续,不依赖单次 Context Window |
| 2 | **工具签名即文档** | Agent 选错工具 |
| 3 | **Sub-Agent 隔离** | 上下文污染 & 工具选择空间爆炸 |
| 4 | **上下游反压(Backpressure)** | 无限循环、错误无法自我修复 |
| 5 | **智能体审智能体** | 自我合理化、偏见在同一 Context 里识别不出 |
| 6 | **熵管理与文档园丁** | 代码库与文档随时间腐化 |

1. **双阶段架构(Initializer + Executor)**:`Initializer 理解任务→写 plan.md→退出` `Executor 读 plan.md→按步执行→跨窗口接力`。两个 Agent **不共享上下文**,只通过 Workspace 里的 `plan.md` 接力 → 任务可跨多次会话延续。
2. **工具签名即文档**:工具名用动词短语(`query_calendar` 而非 `tool_03`);每个参数 description 写清"何时用、何时别用";返回值结构稳定,Agent 不必每次猜格式。**工具描述就是给模型的提示词。**
3. **Sub-Agent 隔离**:关键不是"拆",是"隔离"——每个 Sub-Agent 独立 Context Window、只看到自己需要的工具;主 Agent 只接收**结构化输出**,不接收它的中间思考(噪音)。
4. **上下游反压**:`上游给确定性设置 → Agent 执行 → 下游用 测试/类型检查/Lint/CI 拒收无效工作 → 错误信号回传上游调整`。**Linter 的报错本身就是上下文工程**:不只说"违反规则 X",还解释"为什么有这条规则、正确做法是什么",Agent 读完能自纠,不需人介入。
5. **智能体审智能体**:让另一个 Agent 做 Code Review,但**必须换 Context**——Reviewer 只看 `git diff + rules/*.md`,角色设成"怀疑态度的资深评审",对主 Agent 的产出一无所知,才不会被"自我合理化"污染。**失效的从来不是"换个模型再评",而是"用同样的 Context 再评一次"——那只会复现同一个偏见。**
6. **熵管理与文档园丁**:代码库的熵会随时间增长,**Agent 比人长得更快**(它擅长"模式复制",会忠实放大坏模式)。解法:后台跑一个"文档园丁"Agent 定期扫过期文档、检测架构漂移、提清理 PR;把"垃圾回收"做成定时任务,而非项目末尾的善后。

## 6. 核心部件速查(Harness 里到底有哪些零件)

> 铁律是"心法",模式是"招式",这一节是"零件清单"——一个完整 harness 通常由这些部件拼成:

| 部件 | 一句话职责 | 关键工程要点 |
|---|---|---|
| 工具(Tools) | 给模型装上"手" | 描述即提示词;按需加载 schema,别一次性全摆出来撑爆上下文 |
| 控制循环(Loop) | 决定"再想一步还是收工" | 停机条件、步数上限、被拒/报错如何反馈回去而非硬重试 |
| 上下文管理 | 经营窗口这块稀缺预算 | 裁剪、压缩(compaction)、按需召回 |
| 记忆(Memory) | 窗口之外的长期读写 | 一条事实一处存储 + 索引,每次按需读回(本质 RAG) |
| 子代理(Sub-agent) | 分而治之 + 隔离上下文 | 独立上下文、专属工具、只回汇结构化结果;💰冷启动有成本,别滥用 |
| 权限护栏 | 可控副作用 | 只读/确认/放行;难逆或对外动作默认先确认;一处授权不延续到下一处 |
| 钩子(Hooks) | 把"每次都要做的事"固化 | 由系统强制执行,不指望模型记得(如提交前自动跑 lint) |
| 技能(Skills) | 可复用的工作流封装 | 把专家流程沉淀成资产,跨会话复用 |
| 验证闭环 | 干完 ≠ 干对 | 断言"完成"前先跑验证拿证据;失败如实回报 |

## 7. 一手样本:这篇文档就是在一个 harness 里写出来的

> 我(写这篇笔记的 AI)此刻就跑在一个成熟 harness 内。对照上面的四铁律 / 六模式 / 零件清单,几乎每一项都能在我的运行环境里找到实物:

| 文章里的概念 | 在我当前运行环境里的对应物 |
|---|---|
| 铁律一 · 上下文越少越好 | 对话变长时系统会把前文**压成摘要**再续上(compaction),不无脑堆全历史 |
| 铁律二 · 专才 > 通才 | 可派发只读搜索代理、代码评审代理等专才并行干活 |
| 铁律三 · 状态写文件 | 一个磁盘上的**记忆目录**:一条事实一个文件 + 一个索引,跨会话读回 |
| 铁律四 · 约束可执行 | 某些动作前后被**钩子**拦截,钩子输出当作反馈对待——硬规则由系统强制 |
| 模式 3 · Sub-Agent 隔离 | 子代理带独立上下文、专属工具,只回汇结构化结果 |
| 模式 4 · 反压 | 工具调用被拒/报错会作为信号回灌,让我换方案而非硬重试 |
| 模式 1 · 双阶段 | 可先出"计划"再进入"执行",计划落盘后跨轮接力 |
| 验证闭环 | 规则明确要求:声称"完成/已修复"前必须先跑验证拿证据 |
| 工具签名即文档 | 我能调的每个工具都带详细说明,写明"优先用 X 而非 Y""是否有副作用" |

> 这正说明:**Harness 工程不是理论,而是已经在产品里跑着的工程实践。** 同一个底层模型,套上更好的 harness,就能从"会聊天"变成"能交付"。模型能力是上限,harness 决定你实际拿到上限的百分之几。

## 8. 实战案例:悟空 AI 招聘(专才赢通才的真实样本)

> 来自参考文章中阿里团队在钉钉企业级 Agent「悟空」上的落地。数字均为其内部实测,仅作"改造前 vs 改造后"的相对参考。

**第一版(踩坑):全能招聘 Agent** —— 一份 600+ 行 System Prompt + 13 个工具,什么都包。结果:
* RPA 跑到一半"顺手"去回复候选人聊天 → 上下文炸了;
* 10+ 工具里挑错工具概率显著偏高(该调日历调成了查库);
* Prompt 写到第 400 行,模型已忘了开头说"不要主动发 Offer";
* 状态全在上下文里,换个会话完全不记得这个候选人;
* 出错不知哪环错了,回滚一次查 2 小时日志。
* ——同时违反铁律二(专才>通才)和铁律一(上下文越少越好)。

**第二版(重构):2 Agent + N Skill 专才架构**
```
悟空(Orchestrator:拆任务/分发/维护 Workspace/跨会话记忆)
  ├── 人岗匹配 Agent(RPA 专才,Prompt 80 行,4 工具)
  └── 招聘沟通 Agent(聊天专才,Prompt 90 行,5 工具)
        └── N 个原子化 Skill(parse_resume / score_match / query_calendar / ...)
              └── Workspace(真相之源:/candidates/{id}/state.json 等)
```
* **最关键一招**:悟空不直接调 N 个工具,只调"派单"这一个动作——把任务连同**精挑的 3-5 个工具**派给专才。这就把"50 选 1"的选择题,拆成了几道"5 选 1"。
* **四铁律的落点**:Prompt 严格 <100 行(铁律一);强行只允许 2 个 Agent、其余下沉为 Skill(铁律二);候选人状态/RPA 进度全写 Workspace(铁律三);合规规则("不许主动发 Offer""超时 72h 提醒 HR")写成 Linter、调用前先校验(铁律四)。
* **改造后**:工具选择错率显著下降、端到端准确率稳超上线门槛、日志按 Agent 分桶分钟级定位、上下文消耗回到安全区、新增需求"加一个 Skill 半天上线"。

**三条血泪经验:**
1. **Agent 数量别超 3 个,Skill 可以无限加** —— Agent 数量本身也是上下文成本,堆到第 6 个 Agent,编排层开始"选错 Agent"。
2. **RPA + Agent 的接缝最容易出事,要做"事务边界"** —— 用 `rpa_lock/{batch}.json` 记 running/done,中断后从断点续传,绝不重头跑(对应铁律三)。
3. **对外说话的 Agent 必须接硬护栏** —— 招聘沟通 Agent 套三层:①白名单工具(禁用撤回/群发)②Linter 拦截敏感词/合规 ③第二个 Agent 用独立 Context 审稿。事故率从"每周一两次"降到"记不清上次是什么时候"。

## 9. 行业标杆地图

> 不是排名,是"四根护栏在五层里的落点不一样"。

| 团队 / 产品 | 最有辨识度的 Harness 选择 | 启示 |
|---|---|---|
| Anthropic / Claude Code | Initializer + Executor 双阶段,Workspace 作持久层 | 跨会话延续 = 真生产力,不是花活 |
| LangChain / Deep Agents | 自我验证 + 追踪 + 工具签名优化,不换模型冲进 Top 5 | Harness 回报曲线比换模型陡得多 |
| Mitchell Hashimoto / Ghostty | AGENTS.md 作"项目宪法",每条规则对应一个真实失败 | 能机器化就别留在自然语言 |
| Cursor / Cline | 内置反馈回路(Linter/Type Check/Test)自动闭环 | 反馈回路本身就是上下文,错误信息要写给 Agent 看 |
| 悟空 AI 招聘 | 2 Agent + N Skill + Workspace 状态文件 + Linter 硬护栏 | 企业级:"对外说话"和"动用户数据"必须有硬护栏 |

## 10. 未来趋势(带可证伪条件,非预言)

| 趋势 | 方向 | 可证伪条件 |
|---|---|---|
| 岗位迁移 | Prompt 工程师 → Agent / Harness 工程师(卷的从"话术"变成"系统") | 若 12 个月内 Prompt 工程师岗位重新增长且薪资中位数回升,则需修正 |
| 协作协议 | 工具调用(MCP) → Agent 间协作(A2A),多 Agent 像微服务一样工业化 | 若两年内无主流 SDK 采用的 A2A 草案,则节奏延后 |
| Agent OS | 单 Agent 完成任务 → "Agent 操作系统",竞争焦点从"多智能"变"多稳" | —(趋势性判断) |
| 接口标准化 | 模型与 Harness 接口标准化(工具 Schema/上下文交换格式/Sub-Agent 调用协议) | 换模型像换数据库引擎一样简单——前提是 Harness 写得够干净 |

## 11. 常见误区

| 误区 | 纠正 |
|---|---|
| "换个更强的模型就行了" | 模型是上限,harness 决定你拿到多少。LangChain 不换模型只优化 harness 就 30→5。 |
| "Prompt 写够好就不需要 harness" | Prompt 管单次表达,管不了多步执行的循环、副作用、回滚。 |
| "窗口够大就把全部历史塞进去" | 成本线性涨 + 中段遗忘,反而更差。要主动经营窗口。 |
| "工具越多越能干" | 工具过载会让模型"逛超市"、撑爆上下文。按需加载、精选工具。 |
| "做一个全能 Agent 全包" | 通才跑不过专才。Agent 贵、Skill 便宜,能用 Skill 别加 Agent。 |
| "让 Agent 记住进展" | 上下文是易失的,状态要写文件,Workspace 才是档案。 |
| "把规则写进文档让它读" | 文档是建议,Linter/CI 才是强制。能机器化就别留自然语言。 |
| "模型说做完了就是做完了" | 没有验证证据的"完成"不可信。证据先于断言。 |

## 12. 心法收尾:六句话 + 一个比喻

| # | 心法 | 注解 |
|---|---|---|
| 1 | 你优化的不是 Agent,是 Agent 的**工作环境** | 把它当员工,而非工具 |
| 2 | 上下文是**稀缺资源**,不是无限仓库 | 少即是多,干净比丰富更重要 |
| 3 | 状态写在**文件**里,不在脑子里 | Context 是工位,Workspace 才是档案 |
| 4 | 能写成 **Linter** 的约束,别写成文档 | 机器能强制的,永远比人能记住的可靠 |
| 5 | **Agent 贵、Skill 便宜、护栏最便宜** | 能用 Skill 别加 Agent,能用 Linter 拦别靠 Prompt |
| 6 | **对外说话 / 动用户数据**的地方,硬护栏要早一步到位 | Prompt 可以漏、模型可以错,合规底线不能 |

> **模型不是孙悟空,Agent 系统才是。** 孙悟空再厉害,没有金箍棒、筋斗云、紧箍咒、取经路线图,他就是一只猴子。模型再聪明,没有合适的 Harness,它就是一匹脱缰的野马——能跑也能毁。
> 真正的护城河,是你为这匹野马造的那条高速公路:它决定了你能跑多远、多稳、多快。**Harness Engineering,是 AI 时代软件工程的重新发明。**
