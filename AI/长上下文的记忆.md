# 长上下文的记忆(落地导向)

> 视角:LLM(大语言模型)应用 / Agent(智能体,能自己决定下一步动作的 LLM 程序)的"记忆"到底怎么做 + 长上下文窗口够不够用 + 工程取舍
> 标注约定:🟢 开源/可私有部署　🔴 闭源/API　💰 主要成本项
> 写法约定:术语后用括号补一句人话,看不懂的词不用跳出去查。
> 更新基准:2026-06。模型上下文上限、各家记忆能力变化快,看相对结论即可。
> 关联文档:本篇与 [RAG技术选型.md](RAG技术选型.md) 第 8.1 节(长上下文 vs RAG)是同一组取舍的两面——RAG(检索增强生成,先从库里搜到相关资料再让模型据此作答)解决"从大库里找知识",记忆解决"跨轮次/跨会话记住交互"。

---

## 0. 一句话原则

> **长上下文(context,模型一次能读进去的全部文字)是"工作记忆",不是"长期记忆"。**
> 窗口(window,这次对话能容纳文字量的上限)再大,也只解决"这一次对话塞得下多少",解决不了"关掉再开还记得""一年的交互攒成画像""成本和中段遗忘"。**长期记忆必须是窗口之外的一套读写系统**,长上下文只是它的一段缓存(cache,临时存一份就近取用的数据)。

---

## 1. 先分清三种"记忆"

| 类型 | 类比 | 存在哪 | 生命周期 | 典型内容 |
|---|---|---|---|---|
| **工作记忆(Working,当前正在处理的)** | 人的"当前脑中所想" | **上下文窗口内** | 单次请求 | 当前问题 + 刚拼进去的检索结果/历史 |
| **短期记忆(Short-term,一次会话内的)** | 一次会话的来龙去脉 | 会话状态(常驻 Redis/内存。Redis=一种内存数据库,读写极快但重启易丢) | 单次会话 | 多轮对话历史、临时变量、本轮目标 |
| **长期记忆(Long-term,跨会话沉淀的)** | 笔记本 / 长久印象 | **外部存储**(数据库/向量库/文件/知识图谱) | 跨会话、可累积 | 用户画像(对一个人的长期了解)、偏好、过往结论、事实库 |

> **核心**:长上下文窗口 = 把更多"工作记忆 + 短期记忆"一次性塞进去。**它天然碰不到长期记忆**——长期记忆必须落到窗口之外、并在每次对话时**按需读回**(检索/召回,retrieval,按相关度从库里把相关条目找回来),这一步本质就是 RAG。

---

## 2. 为什么"窗口够大就行"是个陷阱

> 反直觉:即便有 100 万词元(token,模型处理和计费文本的最小单位,中文 1 字 ≈ 1–2 词元)的窗口,也不该把"全部历史"无脑塞进去。四个硬伤:

| 问题 | 说明 |
|---|---|
| **成本线性涨** | 💰 每轮都按"塞进去的全部 token"计费 + 首字延迟(从提交到吐出第一个字的等待时间)变长;长会话里历史越滚越多,每轮重复付费 |
| **中段遗忘(Lost in the Middle)** | 内容堆太多时,模型对**开头/结尾**利用最好,**正中间易被忽略**——塞得多 ≠ 用得到 |
| **跨会话不持久** | 窗口是"一次性"的:会话结束、上下文一清,记忆归零;换设备/换线程也带不走 |
| **窗口仍有上限** | 再大也有边界,且接近上限时**有效利用率下降**(长程注意力衰减:输入越长,模型对远处内容的关注越弱),不是"线性可靠" |

> 结论:长上下文负责**把当前最相关的一小撮信息精准喂进去**;**攒、筛、写回、跨会话取**这些事,要交给记忆系统。和 RAG 一样——**靠精排(把候选重新打分排序、只留最相关的)而非堆量**。

## 3. 记忆系统的标准架构

```
写入(Write):  对话/事件 → 抽取(extract,用 LLM 提炼出要点)→ 去重/合并(consolidate)→ 写入长期存储
读取(Read):   新一轮 query(用户这一轮的问题)→ 召回相关记忆(检索)→ 拼进上下文 → 生成
遗忘(Forget):  按时效/重要度/冲突 → 衰减(降权)、覆盖、淘汰
```

### 3.1 写入:从"原始对话"到"可复用记忆"

- **不要直接存原始对话**:噪声大、检索差。用 LLM 把对话**抽成结构化记忆**(整理成规整的事实/偏好/结论条目),如"用户偏好简洁回答""项目用 PostgreSQL(一种关系型数据库)"。
- **去重与合并(consolidation,把重复/相关的记忆并成一条)**:新记忆要和旧记忆比对——**重复就跳过,冲突就更新/覆盖**(如"用户搬到了上海"应更新旧地址),否则记忆库越长越乱、自相矛盾。
- **打时间戳和来源**:每条记忆带 `时间/会话ID/置信度(有多确信这条是真的)`,方便后续做时效衰减和溯源(回查这条记忆从哪来)。

### 3.2 读取:别每轮全量拼,要召回

- 和 RAG 完全同构(结构一模一样):把长期记忆**向量化入库**(embedding,把每条文本转成一串数字/向量,好算语义相似度),新一轮按 query **召回 Top-K(相关度最高的前 K 条)记忆**拼进上下文,而不是把所有记忆全塞。
- **分层拼接**:系统提示(system prompt,给模型设定的固定人设/规则)+ 召回的长期记忆(画像/事实)+ 短期会话历史(最近 N 轮)+ 当前 query。
- ⚠️ 给记忆也设 **token 预算**(分配给它的字数额度,如占窗口 10–20%),别让历史/记忆挤掉真正的任务上下文。

### 3.3 遗忘:记忆不是只增不减

- **时效衰减**:旧的、低置信(不太确信)的记忆降权或淘汰;时间敏感事实(职位、住址)新值覆盖旧值。
- **重要度分级**:核心画像长期保留,临时事实可过期。
- **冲突消解**:同一槽位(slot,预先定好的字段格子,如"住址"这一格)多条冲突 → 保留最新/最高置信,而非全部召回(否则模型被矛盾信息搞晕)。

## 4. 实现形态(从轻到重)

| 形态 | 做法 | 适用 | 取舍 |
|---|---|---|---|
| **滚动摘要(Summary Buffer)** ⭐ | 会话超长时,把旧历史**总结成一段摘要**替代原文,只保留最近 N 轮原文 | 长会话、最常用的起步方案 | 简单有效;但摘要会丢细节,关键信息要单独抽成记忆 |
| **向量记忆(Vector Memory)** ⭐ | 记忆条目向量化入库,按需召回 | 跨会话长期记忆、用户画像 | 就是 RAG 那套,成熟;需管去重/时效 |
| **结构化记忆(Key-Value / 槽位)** | 把画像存成一个个字段(`偏好/语言/项目栈`),键值对存取 | 字段明确、要精确读写的场景 | 准、可控、好维护;但表达力有限,非结构信息装不下 |
| **知识图谱记忆(Graph)** | 记忆存成"实体-关系"的网(如 张三—住在—上海),支持多跳关联(顺着关系连查几步) | 关系复杂、要推理关联的场景 | 表达力强、能多跳;💰 构建/维护成本高,别开局就上 |
| **分层记忆 + 分页(MemGPT 式)** | 仿操作系统:"主上下文"放热数据(常用的),"外部存储"放冷数据(不常用的),模型自己**换页**(paging,像电脑内存不够时把数据在内存和硬盘间倒腾)调取 | Agent 长程任务、要"无限记忆"错觉 | 强大但复杂,链路长、难调试 |

> 落地优先级:**先滚动摘要 + 向量记忆**(覆盖 90% 需求)→ 画像类信息补**结构化槽位** → 真有复杂关联推理再上**图记忆 / 分层分页**。

## 5. 方案与工具

| 方案 | 形态 | 开源 | 一句话 |
|---|---|---|---|
| **Letta(原 MemGPT)** | 分层记忆 + 分页 Agent | 🟢 | "记忆即操作系统"范式的代表,自管主/外部上下文换页 |
| **mem0** | 记忆层中间件(夹在应用和模型之间的一层) | 🟢 | 给应用加一层"抽取-存储-召回"记忆 API,接入快 |
| **LangGraph / LangMem** | 编排(把多步流程串起来)+ 记忆工具 | 🟢 | 在状态机(把流程画成"节点+转移"的图)里管短期(checkpointer,检查点,存中间状态便于恢复)+ 长期(store,持久存储)记忆 |
| **MCP `memory` server** | 知识图谱记忆 | 🟢 | 通过 MCP(Model Context Protocol,模型上下文协议,给模型挂外部数据/工具的标准接口)给 Claude 等挂一个本地知识图谱记忆,跨会话持久 |
| **各厂商内置记忆** | 产品级 | 🔴 | ChatGPT Memory / Claude Projects 等,免运维但数据在云、可控性弱 |

> ⚠️ 选型提醒(同 RAG):**别一上来就上重框架**。记忆的难点和 RAG 一样在**数据细节**(抽什么、怎么去重、何时遗忘、召回准不准),框架解决的是编排,解决不了这些脏活。POC(概念验证,先搭个小原型证明可行)用框架快,生产阶段核心读写逻辑往往值得自己掌控。

## 6. 长上下文 vs 记忆系统:怎么选

| 维度 | 纯长上下文 | 记忆系统(外部) |
|---|---|---|
| **跨会话持久** | ❌ 关了就没 | ✅ 落库(写进数据库),长期累积 |
| **单轮成本/延迟** | 💰 高,历史越滚越贵 | 低,只召回相关几条 |
| **信息利用** | 长了有中段遗忘 | 精排后只给相关记忆,信噪比高(有用信息占比高) |
| **可控/可编辑** | 弱,记忆混在对话里 | ✅ 记忆可增删改查、可审计(能逐条查到/核对) |
| **实现复杂度** | ✅ 低,塞进去就行 | 高,要抽取/去重/召回/遗忘一整套 |

> **最佳实践是二者结合**:**长上下文当"工作记忆缓存"**——把召回的长期记忆 + 最近几轮原文,一次性给足模型深读;**记忆系统当"长期硬盘"**——负责攒、筛、跨会话取。短会话/一次性任务,纯长上下文足矣;**有"跨天记住用户"诉求,就必须上记忆系统**。

---

## 7. 工程化要点

- **短期 vs 长期分开存**:短期会话状态放 Redis(快、易过期);长期记忆放 PG(PostgreSQL 的简称)/向量库(持久、可检索)。
- **写入异步化(async,不卡住当前流程、后台慢慢做)**:记忆抽取/去重是额外 LLM 调用,💰 别卡在主回复链路上,**回复后异步写回**。
- **可观测(observability,系统跑起来后能看到内部发生了什么)**:记录每轮"召回了哪些记忆、用没用上、用户反馈",和 RAG 一样靠反馈数据持续优化。
- **隐私与可删除**:长期记忆=持续画像用户,**必须支持按用户清除、且尊重"别记住这条"的诉求**;敏感信息(身份证/密钥)入库前脱敏(把敏感内容打码或替换)。
- **评估**:别只看"答得顺不顺"。构造跨会话测试(上一会话告诉它 X,下一会话问它是否记得),量化**记忆召回率(该记得的有没有取回来)**和**冲突/串记率(把张三的事记到李四头上的比例)**。

## 8. 案例:Claude Code 的记忆机制(理论的真实落地)

> 来源:Claude Code 官方文档《Claude 如何记住你的项目》(code.claude.com/docs/zh-CN/memory)。
> 它是本文前 7 节原则的一个**生产级实现**,且刻意做得"轻"——可以当作"记忆系统最小可用形态"来抄。

### 8.1 两套互补的记忆

Claude Code 把记忆拆成两套,**都在每次会话开始时加载**:

| | **CLAUDE.md 文件** | **自动记忆(Auto-memory)** |
|---|---|---|
| 谁写 | **你**(人) | **Claude 自己** |
| 内容 | 指令、规则(编码规范、架构、工作流) | 学到的东西(构建命令、调试经验、你的偏好) |
| 范围 | 项目 / 用户 / 组织(分层) | 每个 git 仓库一份(仓库内 worktree 共享。worktree=git 工作树,同一仓库的多个并行工作副本) |
| 加载 | 每会话**完整**加载 | 每会话只加载索引 `MEMORY.md` 的**前 200 行 / 25KB** |

> ⚠️ 关键性质:这些内容**作为"用户消息"注入,不是系统提示的一部分**——所以是"模型读到并尽量遵守"的**软约束**(尽力而为,不保证),不是硬保证。要**强制**在某时点执行(如提交前必跑 lint,代码风格检查),得用 **hook**(钩子,在固定生命周期事件上自动触发的 shell 命令),而非写进 CLAUDE.md。这正对应第 0 节"记忆是上下文、不是配置"的区别。

### 8.2 存储与加载(印证"分层 + 按需读")

自动记忆落在机器本地一个目录:
```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # 简洁索引(目录),每会话加载(前 200 行 / 25KB)
├── debugging.md       # 详细笔记,启动时不加载
├── api-conventions.md # …按需才用文件工具读
└── ...
```
- `<project>` 来自 **git 仓库**,所以同仓库所有 worktree/子目录**共享一份**;机器本地、不跨设备、不上云。
- **加载策略 = 本文第 6 节的核心**:只把**索引**(`MEMORY.md`)常驻上下文,**主题文件不预加载、需要时才读**(lazy-load,按需加载、用到才读)——就是"别每轮全量拼,要分层 + 按需召回"。
- **写入有取舍**:官方明确"Claude 不是每会话都存,**根据未来是否有用决定记什么**"——对应第 3.1 节的"抽取 + 判断值不值得记"。

### 8.3 CLAUDE.md 的分层与组织(人写的那一半)

- **四级范围**(按加载顺序,范围从宽到窄,越靠后越优先):
  托管策略(组织,IT 下发、不可被个人排除)→ 用户 `~/.claude/CLAUDE.md` → 项目 `./CLAUDE.md` 或 `./.claude/CLAUDE.md` → 本地 `./CLAUDE.local.md`(个人、加 `.gitignore` 不提交)。
- **目录树向上拼接**:从工作目录逐级向上找 `CLAUDE.md`,全部**拼接**(叠加,不是覆盖);子目录的 CLAUDE.md **按需**(读到该目录文件时)才加载。
- **`@path` 导入**:`@README`、`@docs/git.md` 会在启动时展开进上下文(最多递归 4 跳);想"提到路径但不导入"就用反引号把它包起来。
- **`.claude/rules/`**:把规则拆成多文件;带 `paths:` frontmatter(写在文件顶部的元数据配置块)的**路径范围规则**只在 Claude 处理匹配文件时才加载 → 省上下文(同第 6 节"按需,不要堆")。
- **大小纪律**:单个 CLAUDE.md **目标 < 200 行**,越长越费上下文、遵守度越低。注意 `@import` 只帮组织、**不省上下文**(导入文件启动时一并加载);真要省得用路径范围规则。

### 8.4 操作入口与生命周期

- **`/memory`**:列出本会话加载了哪些 CLAUDE.md/规则文件、开关自动记忆、打开记忆文件夹。
- **`/init`**:自动分析仓库生成起始 CLAUDE.md(构建命令、约定、架构)。
- **让它记**:你说"以后都用 pnpm" → 进**自动记忆**;你说"加到 CLAUDE.md" → 才进 CLAUDE.md。
- **`/compact` 存活**:`/compact`(把变长的对话压缩成摘要以腾出窗口)后,项目根 CLAUDE.md 会从磁盘**重新注入**;子目录嵌套的要等下次读该目录才重载——所以"只在对话里说过、没落盘(没写进文件)"的指令压缩后会丢,**要持久就写进文件**。

### 8.5 它和本文档的对照

| 本文原则 | Claude Code 的实现 |
|---|---|
| 长上下文=工作记忆,长期记忆落窗口外(第 0/1 节) | 对话窗口是工作记忆;`memory/` 目录是长期记忆,关了会话还在 |
| 别全量拼,索引常驻 + 主题按需读(第 6 节) | `MEMORY.md`≤200 行常驻,`debugging.md` 等需要时才读 |
| 写入要抽取、判断值不值得记(第 3.1 节) | "根据未来是否有用决定记什么",不每会话全存 |
| 三层记忆(第 1 节) | 上下文窗口 / 会话状态 / `memory/` 目录,一一对应 |
| 可审计、可删除(第 7 节) | 全是纯 markdown,`/memory` 可看可改可删 |

> 💡 **一个值得抄的取舍**:官方机制目前是**"按需读取整个主题文件"(标量式 lazy-load:直接按文件名读整个文件,不算语义相似度),而不是向量召回 Top-K**——因为记忆文件就几个,直接读文件比建向量库更简单划算。这正是第 4/6 节"先轻后重、别为量小过度设计"的体现;等记忆涨到几百上千条、读文件都嫌多时,才该升级成向量召回(第 4 节的"向量记忆")。

## 9. 常见坑

- **只增不减**:记忆库无限膨胀、自相矛盾 → 召回一堆冲突信息把模型搞晕。**遗忘/合并和写入一样重要。**
- **存原始对话当记忆**:噪声大、召回差。**先抽成结构化事实再存。**
- **每轮全量拼历史**:既贵又触发中段遗忘。**要召回,不要堆。**
- **把长上下文当长期记忆**:换会话就发现"它什么都不记得"。窗口不是硬盘。
- **串记 / 张冠李戴**:多用户/多会话记忆没隔离(没分开)→ A 的偏好用到 B 身上。**记忆必须按用户/会话强隔离 + 召回后校验归属(确认这条确实属于当前用户)。**
