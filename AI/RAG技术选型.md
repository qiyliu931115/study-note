# RAG 技术选型(落地导向)

> 视角:可落地的具体方案 + 成本 + 部署 + 工程取舍
> 标注约定:🟢 开源/可私有部署　🔴 闭源/API　💰 主要成本项
> 更新基准:2026-06。榜单分数多为厂商自报,看相对位置即可。

---

## 0. 一句话选型原则

> **先搭一条可评测的最小闭环,用评测集定位瓶颈,再针对性替换组件。**
> 不要一上来堆 GraphRAG(把文档建成知识图谱再检索)、多路召回(同时跑多种检索再合并)这些复杂架构。90% 的效果问题出在**解析质量**(文档没解析对)和**重排序(rerank,给搜索结果重新排序)**两处,而不是架构不够花哨。

---

## 1. 整体架构与选型总览

```
离线入库(Ingestion):  解析 → 清洗 → 分块(chunking)→ 向量化(embedding)→ 建索引(向量+关键词+元数据)
在线查询(Query):       查询改写 → 混合召回 → 重排序(rerank)→ 上下文组装 → 生成(带引用)
```

| 环节 | 默认推荐(落地) | 备选 | 关键取舍 |
|---|---|---|---|
| 文档解析 | 🟢 MinerU 2.5 / PaddleOCR-VL | 🔴 Mistral OCR(免运维) | 自部署 VLM 解析模型→准,但要养 GPU;调 OCR API→免运维,但按页计费,且需核对 license 能否商用 |
| 分块(chunking) | 结构感知 + 父子块(Parent-Child) | 语义分块 | 块太小→检索准但上下文缺、答不全;块太大→上下文全但向量被稀释、检索糊。用**父子块**(小块检索、大块喂模型)兼顾两头 |
| 向量化(embedding) | 🟢 BGE-M3 / Qwen3-Embedding | 🔴 OpenAI text-embedding-3 | 数据敏感→私有化部署;否则调 API 更省事。**换模型会改变向量维度(如 768→1024)→必须全库重灌**(全部文档重跑 embedding 重新入库),故须提前定死 |
| 向量库 | 🟢 pgvector(中小)/ Milvus(大) | 🔴 Pinecone | 向量量级不大别上 Milvus 这类分布式库;已有 Postgres 直接加 **pgvector** 插件,省一整套独立运维 |
| 检索 | 向量 + BM25 混合(RRF) | 纯向量 | 纯向量(语义检索)会漏掉编号、型号等专有名词→需叠加 **BM25**(关键词检索)做**混合召回**兜底 |
| 重排序(rerank) | 🟢 BGE-Reranker-v2 | 🔴 Cohere Rerank | 用 **cross-encoder**(交叉编码器)对召回结果重新打分排序,算力代价小、准确率提升大,**几乎必做** |
| 生成大模型(LLM) | 🔴 Claude / GPT　🟢 Qwen3 私有 | — | 云端 API(Claude/GPT)质量高,但数据要出域;私有部署(Qwen3)数据不外泄,但自担 GPU 成本 |
| 评估 | RAGAS + 自建问答(QA)集 | TruLens | **检索与生成分开评**:Recall@k 看"有没有搜到",忠实度(faithfulness)看"搜到了有没有用好",否则定位不了瓶颈 |

## 2. 文档解析层(幻觉最大诱因,优先投入)

**范式:已从「OCR+版面拼装」转向「视觉语言模型(VLM)端到端」。**
- 老办法:先 OCR 认字、再用版面模型判断标题/表格/段落、最后拼起来,**多个模型串联、误差层层累积**。
- 新办法:**端到端**——直接把整页图片喂给一个 VLM,一步输出带结构的 Markdown(谁是标题、哪是表格,模型自己判断)。
- 2026 开源模型已追平/反超闭源。

| 方案 | 是否开源 | 收费方式 | 部署方式 | 硬件成本 | 落地建议 |
|---|---|---|---|---|---|
| **PyMuPDF4LLM**(快通道) | 🟢 开源(AGPL/商用授权) | 免费自托管 | 本地库,无模型 | 几乎为零(纯 CPU) | 数字版 PDF(有文字层)首选,先抽再判断要不要上 VLM |
| **MinerU 2.5 / v3.1** | 🟢 开源 ⚠️ 许可证(license)偏严 | 免费自托管 | 私有部署(GPU) | 1×16–24G GPU | 中文长文档首选,跨页表格强;**商用前务必核对授权** |
| **PaddleOCR-VL** | 🟢 Apache 2.0 | 免费自托管 | 私有部署(GPU) | 1×16G GPU(~1.6B) | **商用最友好**,许可证无忧,性价比最高 |
| **Granite-Docling-258M** | 🟢 Apache 2.0 | 免费自托管 | 本地/CPU 即可 | CPU 可跑(~3s/页),或入门级 GPU | 资源受限/边缘部署,结构最干净 |
| **GLM-OCR** | 🟢 开放权重(查许可证) | 免费自托管 | 私有部署(GPU) | 1×GPU | 精度黑马(OmniDocBench v1.5 ~94.6),追求 SOTA 时 |
| **Mistral OCR 3 / Reducto / LlamaParse** | 🔴 闭源 | 💰 按页计费(API) | SaaS,无需部署 | 无(对方承担) | 不想运维 GPU、文档量不大时;量大长期不划算 |
| **AWS Textract / Google Document AI / Azure DI** | 🔴 闭源 | 💰 按页计费(API) | 云服务 | 无 | 已有对应云栈、需合规背书时的稳妥选择 |

**工程取舍:分层路由**
- 数字版 PDF(有文字层)→ 快速通道(PyMuPDF4LLM)直接抽,几乎零成本
- 扫描件/复杂版面/表格密集 → 才上 VLM
- ⚠️ VLM 有**幻觉风险**(脑补文字),关键文档需抽检校验
- 表格单独转 Markdown / 做摘要(summary),否则检索和生成都会崩

## 3. 分块策略(决定召回上限)

> 核心矛盾:**分块(chunk)太小** → 语义不完整、命中了也答不全;**分块太大** → 噪声稀释向量、检索不准、喂给 LLM 浪费词元(token)。
> 分块的目标是让「**检索单元**」尽量精准,同时让「**喂给 LLM 的上下文单元**」尽量完整——这两者**不必是同一个分块**(见 3.2)。

### 3.1 基础切分方法(怎么把文档切成块)

| 方法 | 原理 | 适用 | 取舍 |
|---|---|---|---|
| **固定长度切分** | 按词元(token)/字符定长 + 重叠(overlap) | 基线(baseline)、结构差的纯文本 | 简单粗暴,易切断句子/语义,只作兜底 |
| **递归分块(recursive)** | 按分隔符优先级递归切(段落→句→词) | 通用默认,LangChain 标配 | 比定长好,但不懂文档语义 |
| **结构感知分块** ⭐ | 沿解析出的标题层级/段落/表格切,不跨章节边界 | **首选**,长文档/有版面结构 | 效果最好,但**强依赖解析质量**(见第 2 节) |
| **语义分块(semantic)** | 算相邻句向量(embedding)相似度,在"断点"切 | 无清晰结构的长纯文本 | 慢、要额外向量化调用,**效果不一定更好,别盲目上** |
| **大模型/命题分块(agentic)** | 用 LLM 把文本拆成自洽的"命题/语义单元" | 高价值、低频、追求极致召回 | 💰 贵且慢,只在小规模高价值库用 |

> 落地默认:**结构感知分块**为主,解析质量不行的部分用**递归分块**兜底。语义/大模型分块作为后期优化备选,不要开局就上。

### 3.2 高级检索单元(检索块 ≠ 上下文块,长文档关键)

| 策略 | 做法 | 解决什么 |
|---|---|---|
| **父子块(Parent-Child / Small-to-Big)** ⭐ | 用**小块**(句/小段)做向量精准命中,返回时给 LLM 它所在的**父块**(整段/整节) | 长文档 RAG 最实用技巧:召回准 + 上下文全 |
| **句窗口(Sentence-Window)** | 用单句做检索,返回时带上前后 N 句窗口 | 同上的轻量版,适合常见问答(FAQ)型 |
| **上下文增强检索(Contextual Retrieval)** ⭐ | 入库前用 LLM 给**每个分块生成一句"它在全文中的上下文"**(如"本段出自 X 公司 2024 Q3 财报,讲营收")拼到分块前再做向量化 + BM25 | 解决分块脱离原文后"指代不明/缺背景"导致召回失败,实测显著降低召回失败率;💰 一次性入库成本,可用提示词缓存(prompt cache)压价 |
| **层级摘要树(RAPTOR)** | 对分块递归聚类 + 摘要,建多粒度树,既能答细节也能答"全局型"问题 | 跨段归纳、"整篇讲了什么"类问题 |

> 落地优先级:**先上父子块(Parent-Child)**(收益大、实现简单)→ 召回仍不稳再上 **上下文增强(Contextual Retrieval)** → 出现全局型问题再考虑 **RAPTOR**。

> 💡 **实战:父子块的工业实现("连坐召回")**
> 某团队近万份文档的生产做法,是把"父子块"落成两步检索,关键是**第二步不走向量、走标量索引**,所以快、不成瓶颈:
> 1. 向量 `search` 拿 Top-K 切片 → 提取去重后的 `filename` 列表;
> 2. 用标量过滤 `expr: filename in [...]` 把命中文档的**全部切片拉出**,按 `chunk_index` 排序后重拼给 LLM。
> 效果:"搜到了但答错"基本消失。
> (**标量字段** = 文件名、序号这种普通字段,区别于"向量字段";按它过滤走的是传统索引、不算向量相似度,所以快。前提是 schema 里 `filename`/`chunk_index` 建成标量字段,见第 5 节。)

> ⚠️ **实战反例:上下文增强 ≠ 给每块拼同一份全文摘要("摘要污染")**
> 同一团队曾给**每个切片头部都附加同一份全文摘要**,结果摘要块语义"太全能",在 Top-K 常年霸榜,真正含答案的正文切片被挤出去 → 出现"召回对了但答'未找到相关信息'"。
> **教训**:上下文增强加的必须是**每块各不相同的局部上下文**(如"本段讲 X 配置参数"),**绝不能是所有块共享的同一段全文摘要**,否则制造同质化、污染检索。

### 3.3 参数与中文注意事项

- **分块大小**:通用 300–600 词元(token);问答型可更小(150–300),长文归纳型可更大(800–1000)。
- **重叠(overlap)**:10–20%,避免句子/语义被切断;结构感知切分时可更小。
- **按词元切,不按字符**:中文 1 字 ≈ 1–2 词元,用分词器(tokenizer)计数,别用字符数硬估。
- **中文断句**:递归分隔符要包含中文标点(`。！？；\n`),否则会在英文逗号乱切。
- **向量模型有最大长度**(如 512 词元),分块别超过,否则被截断丢信息。

### 3.4 特殊内容单独处理(否则检索/生成都会崩)

| 内容 | 处理 |
|---|---|
| **表格** | 转 Markdown/HTML 整体作一个分块;大表可"每行+表头"或生成摘要(summary)再入库 |
| **代码** | 按函数/类边界切,不要从中间断 |
| **列表/条款** | 保持逻辑完整,带上所属标题;法律条款常需"条"为单位 |
| **图/公式** | 公式转 LaTeX;图用图注(caption)或 VLM 生成描述入库 |

### 3.5 元数据设计(召回精度的隐藏杠杆)

**必带**:文档标题、章节路径(面包屑)、页码、来源、时间、文档类型。

- **过滤检索**:先按时间/类型/部门过滤,再向量召回,精度速度双赢。
- **引用(citation)**:回答里标来源分块 → 可溯源、降幻觉。
- **章节路径拼进分块**:把"文档名 > 章 > 节"前缀加到分块文本里,等于轻量版上下文增强,几乎零成本。

### 3.6 怎么验证分块好不好

分块没有"标准答案",**必须用检索评测集调**:固定向量模型/重排序器,只改分块策略,看 **Recall@k 变化**。先把分块调到召回稳定,再去优化下游。

> 提醒:1 万篇长文 ≈ 数百万分块 → 向量库毫无压力,**别因"量大"过度设计**;真正要花时间的是切分质量,不是存储。

## 4. 向量化模型(Embedding)

| 模型 | 是否开源 | 收费方式 | 部署方式 | 硬件成本 | 落地建议 |
|---|---|---|---|---|---|
| **BGE-M3** | 🟢 开源(MIT) | 免费自托管 | 私有部署(GPU/CPU) | ~560M,小卡甚至 CPU 可跑 | 中英+多语种,一个模型同时产出稠密向量(管语义)、稀疏向量(管关键词,相当于内置 BM25)、多向量(更细粒度匹配),**私有化首选** |
| **Qwen3-Embedding** | 🟢 开源(Apache 2.0) | 免费自托管 | 私有部署(GPU) | 0.6B/4B/8B 可选,按精度选卡 | 中文强,与 Qwen 生态一致;小规格性价比高 |
| **OpenAI text-embedding-3-large** | 🔴 闭源 | 💰 按词元(token)计费 | SaaS,无需部署 | 无(对方承担) | 省心、效果稳,数据可出域时起步快 |

**取舍**:数据不能出域 → 必走开源私有化(BGE-M3 / Qwen3-Embedding)。领域术语多(法律/医疗)→ 考虑领域微调(fine-tune),收益往往大于换更大模型。
**重要约束**:向量化模型一旦选定,**换模型必须全库重新向量化**(数百万分块),迁移成本高 → 选型阶段就用评测集定好,别频繁换。

## 5. 向量数据库

| 方案 | 类型 | 适用规模 | 取舍 |
|---|---|---|---|
| 🟢 **pgvector** | 开源 | 百万级分块 | **已有 Postgres 栈强烈优先**,省一套运维;**HNSW**(一种主流的"近似最近邻"索引算法,牺牲一点点精度换大幅检索提速)索引足够 |
| 🟢 **Milvus / Qdrant** | 开源 | 千万级+ | 需独立部署运维,功能更全(过滤、混合检索) |
| 🔴 **Pinecone** | 托管 | 任意 | 💰 免运维但贵,数据出域 |

**结论**:1 万篇长文(数百万分块)用 **pgvector 完全够**,不要为了"高大上"上 Milvus 集群增加运维负担。元数据过滤 + 向量检索能同库完成是 pgvector 的优势。

> 💡 **实战:schema 设计要倒推检索策略(连坐召回的地基)**
> (**schema** = 表结构定义,即这张"表"里有哪些字段、各是什么类型;Milvus 里一组数据叫一个 **collection**,相当于关系库里的一张表。)
> 要做父子块/连坐召回,**schema 必须提前为它埋字段**,否则后期想做才发现过滤不了:
> - `filename`(或 doc_id)、`chunk_index` 单独建成**标量字段** → 才能用标量过滤高效拉全篇切片、并按序重拼;
> - `content` 存原文、`embedding` 存向量、`id` 自增主键。
> 一句话:**先想清楚检索怎么查,再定 schema**,别只塞 id+content+embedding。

> 📌 **HNSW 标准参数(照着调就对)**:`M=16`、`efConstruction=256`;查询 `ef` 动态调——召回阶段开大(如 128)保召回,精排后缩小(如 64)控延迟。这是社区公认的稳妥默认值,无需过度纠结。

> ⚠️ **实战反例:Chroma 嵌入式 POC 看着快,上量后被迫迁 Milvus**
> 某团队 POC 用 Chroma(pip 装上就跑),进入生产时因**并发查询弱、collection 管理弱**被迫迁 Milvus,而 Milvus 要预定义 schema(不像 Chroma 自动推断),等于**重做一遍数据建模,代价不小**。
> **教训**:① 向量库一旦上量,迁移成本是实打实的,选型阶段就要考虑并发与运维;② 其实该团队数据量很小(短文整篇不切,向量仅 1–2 万条),**若 POC 直接用 pgvector**(并发/事务/标量过滤/运维都成熟),很可能一步到位、根本不用迁。印证上面"别为预期增长过早上重运维"。

## 6. 检索策略

1. **混合召回(必做)**:向量检索(按语义找)+ BM25(按关键词找)两路并行 → 用 **RRF**(倒数排名融合,一种把两路结果按名次加权合并的算法)合成一个榜。
   - 原因:向量(embedding)对**编号、缩写、专有名词**这类"字面要精确匹配"的词召回差,靠 BM25 关键词检索补足。
2. **元数据预过滤**:先按时间/类型/部门把范围缩小,再在小范围里做向量检索,既准又快。
3. **查询改写**:把用户口语化的问题改写得更适合检索,提升召回。常见两招:
   - **HyDE**(假设性文档):先让 LLM 根据问题"编"一段假想答案,再拿这段去检索(假答案和真文档语义更接近,更好命中);
   - **子问题分解**:把一个复杂问题拆成几个小问题分别检索再汇总。
   - 💰 都要多花一次 LLM 调用,按场景开启。

## 7. 重排序(Rerank)——性价比最高的一步

| 方案 | 类型 | 建议 |
|---|---|---|
| 🟢 **BGE-Reranker-v2-m3** | 开源 | 默认首选,私有部署,交叉编码器(cross-encoder)精排 |
| 🔴 **Cohere Rerank 3** | API | 不想部署、效果稳,💰 按调用计费 |

**做法**:先粗召回 top-50(宁可多召、别漏)→ 用**重排序器(reranker)**逐条把"问题 + 候选"一起送进模型打分,精排出最相关的 top-5 → 只把这 5 条喂给 LLM。
**为什么有效**:粗召回的向量检索是"问题和文档各自算向量再比对",较粗;reranker 是把问题和文档**拼在一起**精读打分(交叉编码器 cross-encoder),自然更准。
**这是投入产出比最高的优化**,几乎必上。代价:每次查询多 ~50–200ms。

## 8. 生成层(大模型 LLM)

| 选择 | 类型 | 适用 |
|---|---|---|
| 🔴 **Claude (Opus/Sonnet) / GPT** | API | 质量优先、可上云;长上下文 + 引用能力强 |
| 🟢 **Qwen3 / DeepSeek 私有部署** | 开源 | **数据不能出域**、合规要求高;💰 自担 GPU 但无词元(token)费 |

**提示词工程(prompt engineering,降幻觉关键)**:
- 强约束"只基于提供的上下文回答,无依据就说不知道"
- **强制带引用(citation)**(标注来源分块),可溯源
- 控制上下文词元(token)预算,靠精排而非堆量

**取舍**:合规允许 → API(质量高、免运维);数据敏感/高并发降本 → 私有化开源模型。

---

## 9. 评估(别省,定位瓶颈靠它)

- **检索层**:看"该搜的有没有搜到"。指标:**Recall@k**(正确文档有没有进前 k 名)、**MRR**(正确答案排在第几名,越靠前越好)、命中率 —— 需构造问答(QA)评测集(可用 LLM 半自动生成 + 人工校)。
- **生成层**:看"搜到了有没有答好"。指标:**忠实度(faithfulness,答案是否严格基于检索内容、有没有瞎编)**、答案相关性、引用正确性 —— 用 **RAGAS** 框架 + 人工抽检。
- **核心纪律**:检索和生成**分开评**。答错时先定位是"根本没召回到"(检索的锅)还是"召回了但 LLM 没用好"(生成的锅),再对症下药。

## 10. 工程化

| 维度 | 落地做法 |
|---|---|
| **增量更新** | 文档级哈希(hash),只重处理变更;支持删除/版本化 |
| **流水线(pipeline)** | 解析/向量化做成可重试、可断点续跑;量大用消息队列并行 |
| **成本** | 💰 离线向量化是一次性批处理成本可控;在线成本主要在 LLM 词元(token),靠小块检索+精排压上下文 |
| **延迟** | 改写/混合召回/精排串联,目标 < 1–2s;重排序器(reranker)和 LLM 是大头 |
| **可观测** | 记录每次查询(query)的召回结果、精排分、最终引用,便于复盘 |

## 11. 手搓 vs 框架(LangChain / LlamaIndex)

> **LangChain / LlamaIndex** = 把 RAG 各环节(加载、分块、检索、调 LLM)封装好的开源框架,`pip` 装上就能拼出一条链路;**手搓** = 不用框架,自己把核心链路一行行写出来。

**核心结论:POC 用框架快起步;进生产 + 要长期调数据策略时,把核心链路手搓重写。**

| 维度 | 框架(LangChain/LlamaIndex) | 手搓 |
|---|---|---|
| **上手速度** | 快,几天拼出 demo | 慢,要自己写加载/分块/检索 |
| **控制力** | 弱,改分块/召回逻辑常要和框架封装"对抗" | 强,每个环节都在自己手里 |
| **出问题定位** | 难,报错穿过好几层抽象 | 易,直接断点到具体环节 |
| **适用阶段** | POC / 快速验证业务价值 | 生产 / 需反复迭代数据策略 |

**为什么生产阶段值得手搓**:RAG 的难点几乎全在**数据处理的细节**(分块策略、父子块/连坐召回、元数据增强、摘要污染这类坑),而框架的抽象恰恰把这些环节包了一层,你想精细调反而被挡路。某团队近万份文档的生产系统,**核心 RAG 引擎 + 清洗管道仅约 3000 行手搓代码**,不算多,却换来"出问题能直接定位到具体环节"的可控性。

> ⚠️ **认知提醒:换模型是最常见的"假动作"**
> 效果不行时,团队本能反应是"换个更强的模型"——因为换模型最简单(改个配置)。但**根因往往在数据**(没清洗干净、分块不对、元数据缺失),换再好的模型也救不了。开源框架 `pip install` 跑通 demo 很容易,**制造了"知识库是低门槛"的假象**;demo 到生产之间,隔着所有"框架不会教你的脏活"。

## 12. 多跳推理(Multi-hop)——普通 RAG 的天然短板

> **多跳推理(Multi-hop)** = 答案不在某一段里,要"先查到 A、再用 A 去查 B"分好几步才能得出。普通 RAG 只有**一轮检索**,够不着。

**举例**:问"我们公司 CEO 的老家在哪个省?",知识库里是两份文档:
- 文档甲:"公司 CEO 是张三。"
- 文档乙:"张三出生于杭州。"

人会**分两跳**:CEO→张三(第一跳)→杭州(第二跳)。但普通 RAG 是**一次性检索**:拿"CEO 老家在哪"去搜,文档乙里**没有"CEO"这个词**、语义也不接近 → 搜不到;只搜到讲"CEO 是谁"的文档甲 → **答不出来**。

**卡点**:第二步该搜什么(搜"张三"),取决于第一步的答案,而**单轮检索拿不到这个中间结果**,推理链就断了。

**解决思路(知道即可,出现这类需求再上)**:

| 方案 | 一句话 |
|---|---|
| **多步检索 / Agentic RAG** ⭐ | 让 LLM 像人一样**分步走**:先搜到"CEO=张三",再**自动发起第二次检索**"张三 老家"。本质是给 RAG 加"边查边想"的循环,最主流 |
| **查询分解** | 把复杂问题先拆成几个子问题分别查(第 6 节"子问题分解"的延伸) |
| **GraphRAG** | 提前把文档抽成"实体-关系"知识图谱(张三—是—CEO,张三—出生于—杭州),沿图谱的边一跳跳走,天生适合多跳 |

> ⚠️ 别一上来就上 GraphRAG/Agentic RAG,**复杂度和成本陡增**。先确认业务里真有"答案要串联多篇文档"的多跳问题,普通 RAG 确实够不着,再升级。

## 13. Agentic RAG

> **Agentic RAG** = 给 RAG 装一个"会思考、能自己决定下一步"的 LLM 智能体(Agent),把原来**写死的一条直线**(检索→生成)变成**模型自驱的循环**(想→查→评估→再查/再想→答)。

- **普通 RAG(被动)**:`用户问 → 检索一次 → 塞给 LLM → 生成`,流程固定,LLM 只在最后"读材料写答案"。
- **Agentic RAG(主动)**:LLM 当"指挥官",自己决定——**要不要检索**(简单问题直接答)、**搜什么/要不要先拆解问题**、**结果够不够好**(不够就换查询重搜)、**该查哪个库或工具**、**答案有没有幻觉**(不靠谱就回炉)。

**直观对比**——问"对比 A 产品和竞品 B 的售后政策差异":
- 普通 RAG:整句搜一次 → 可能只搜到一方 → 答得残缺;
- Agentic RAG:先**拆成两个子任务**分别搜 A、B → 各自拿全 → 再**综合对比**输出。

**衍生范式**(都属于 Agentic 思路):**Self-RAG**(模型自判要不要检索、结果可不可信)、**Corrective RAG / CRAG**(检索差时自动改写查询或转搜网页兜底)、**ReAct 式**(边推理边检索)。

**代价(别滥用)**:① 慢、贵——一次问答可能触发多轮 LLM 调用 + 多次检索;② 不稳定——Agent 自主决策偶尔绕路或死循环,需加最大步数限制和兜底;③ 难调试——链路变长。

> 工程实现:常用 **LangGraph**(把"边查边想"的循环/分支画成状态机的编排框架)来搭。**简单问答别上,确认普通 RAG 真不够用再升级。**
